# ドキュメント作成スキル

ユーザーから提供された資料や情報をもとに、`docs/` ディレクトリにドキュメントを作成する。

## 入力

`$ARGUMENTS` に以下のいずれかが含まれる:

- ドキュメントのテーマやタイトル
- 参照URL
- 説明してほしい技術や概念

## 実行手順

1. **目的の確認**: 誰が、何のために読むドキュメントか確認
2. **調査**: 参照URLがあれば WebFetch で内容を確認
3. **構成提示**: ドキュメントタイプに応じた構成案を提示
4. **作成**: 承認後、ドキュメントを作成
5. **調整**: 必要に応じてユーザーと表現を調整

## 出力先

- `docs/` ディレクトリ配下に作成
- Markdown 形式（`.md`）
- ファイル名は内容を端的に表す英語（例: `ssh-agent.md`）

## ドキュメントタイプと構成

目的に応じて適切な構成を選択する（Diataxis フレームワーク参照）。

### 1. 概念説明（Explanation）

「〜とは何か」「なぜ〜なのか」を説明する。

```
# タイトル
## TL;DR（3行以内）
## 概要
## 仕組み（図を含む）
## ユースケース
## 参考資料
```

### 2. 手順書（How-to）

特定のタスクを達成する方法を示す。

```
# 〜する方法
## 前提条件
## 手順
## トラブルシューティング
## 参考資料
```

### 3. リファレンス

設定値、コマンド、APIなどの一覧。

```
# タイトル
## 概要
## 一覧（表形式）
## 参考資料
```

## 作成ルール

### 構造

- **逆ピラミッド**: 最重要情報を最初に置く
- **スキャナブル**: 見出し、箇条書き、表を活用
- **1セクション1トピック**: 情報を適切に分割

### コード例

- **コピペ可能**: そのまま実行できるコードを提供
- **コメント最小限**: コードの説明は本文で行う
- **実際の値**: プレースホルダより具体例を優先

### 図解

- mermaid を使用（環境依存しない）
- flowchart または sequenceDiagram を適切に選択
- 複雑すぎる図は避ける（要素は7個以下）

### 表現

- 技術ドキュメントとして適切な表現
- 日本語ベース、技術用語は英語可
- 曖昧な表現を避ける（「など」「いろいろ」）

### 参照資料

URLを記載する際は、将来リンク切れになっても内容がわかるよう引用文を併記:

```markdown
**参考**: [タイトル](URL)

> 引用文（英語の場合は日本語訳も併記）
```

## AI Agent Context

ドキュメント末尾にHTMLコメントとして記録（将来の更新時の参考用）:

```markdown
<!-- AI Agent Context
作成日: YYYY-MM-DD
目的: なぜこのドキュメントを作成したか
参照: 確認した資料のURL
注意: 将来の更新時に注意すべき点
-->
```
