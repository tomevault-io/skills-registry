# Agentic Development Methodology

> 來源：Andrew Ng - DeepLearning.AI Agentic AI 課程
> 核心理念：**迭代優於一次性** — LLM 透過思考、研究、修訂的工作流程，能顯著超越 Zero-shot 產出品質

---

## 1️⃣ 迭代工作流程 (Iterative Workflow)

### 原則
傳統 LLM 使用 = 「一次性寫完論文，不能使用退格鍵」
Agentic 工作流程 = 允許模型思考 → 研究 → 修訂 → 改進

### 實踐規則

```
❌ WRONG: 單次 prompt → 直接使用輸出
✅ CORRECT: prompt → 初稿 → 反思 → 修訂 → 驗證 → 最終版
```

| 策略 | 說明 |
|------|------|
| **拆解任務** | 複雜任務拆成多步驟（先大綱 → 再搜尋 → 最後寫作） |
| **平行處理** | 獨立子任務同時執行（如多個搜尋同時進行） |
| **模組化** | 各步驟使用最適合的工具/模型，可獨立替換 |

### 開發應用
- **程式碼生成**：生成 → 執行 → 讀取報錯 → 修正 → 驗證
- **文檔撰寫**：主題分析 → 大綱 → 分段撰寫 → 整合 → 審校

---

## 2️⃣ 反思設計模式 (Reflection Pattern)

### 核心概念
要求 LLM **檢查自己的輸出**，找出錯誤或改進空間，撰寫更好版本。

### 兩種反思類型

| 類型 | 描述 | 效果 |
|------|------|------|
| **自我反思** | LLM 審視自己的輸出 | 🟡 有效但有限 |
| **外部回饋反思** | 結合客觀外部資訊 | 🟢 強力提升 |

### 外部回饋來源
```
1. 程式碼執行結果 → Error logs / stdout
2. 工具驗證 → 字數統計、搜尋引擎驗證事實
3. 測試結果 → Unit test pass/fail
4. 用戶回饋 → Human-in-the-loop
```

### 實踐模板
```
[生成] → [執行/驗證] → [收集外部回饋] → [分析錯誤] → [修正] → [重新驗證]
```

### 開發應用
- **Bug 修復**：執行 → 收集報錯 → 餵回 LLM → 生成修正 → 重新執行
- **程式碼審查**：生成 → lint/test → 將結果餵回 → 修正

## 3️⃣ 工具使用 (Tool Use & Function Calling)

### 原則
LLM 不直接執行函式，而是**輸出工具調用請求**，由系統執行後回傳結果。

### 工具類型

| 類型 | 範例 | 用途 |
|------|------|------|
| **資訊獲取** | getCurrentTime, 搜尋 API | 解決幻覺問題 |
| **精確計算** | Calculator, 程式碼執行 | 避免數學錯誤 |
| **外部操作** | API 呼叫, 資料庫寫入 | 擴展能力邊界 |
| **驗證工具** | 字數統計, Schema 驗證 | 提供客觀回饋 |

### 程式碼執行作為工具
```
LLM 寫 Python 程式碼 → 執行 → 返回結果 → LLM 解讀
```
**比直接讓 LLM 預測答案準確得多**

### MCP (Model Context Protocol)
開放標準，構建一次連接器即可讓 AI 存取多種數據與工具（Slack, GitHub, Drive）。

## 4️⃣ 評估與錯誤分析 (Evals & Error Analysis)

> ⚠️ **這是區分高效能團隊與普通團隊的關鍵**

### 核心原則：先構建，再分析
```
❌ WRONG: 花數週空想完美架構
✅ CORRECT: 快速原型 → 觀察失敗 → 分析根因 → 迭代改進
```

### 錯誤分析方法

#### 1. 檢查軌跡 (Traces)
不能只看最終結果，必須檢查**中間步驟**：
- 搜尋關鍵字是否正確？
- 搜尋引擎是否返回了垃圾網站？
- 寫作模型是否忽略了重點？

#### 2. 量化數據
建立試算表，統計**每個組件**導致錯誤的頻率：
```
| 組件 | 錯誤次數 | 佔比 | 優先級 |
|------|---------|------|--------|
| RAG 查詢 | 75 | 75% | 🔴 HIGH |
| LLM 推理 | 15 | 15% | 🟡 MED |
| 工具調用 | 10 | 10% | 🟢 LOW |
```

### 評估矩陣 (2x2)

|  | 程式碼/客觀 | LLM 評審/主觀 |
|--|------------|--------------|
| **有標準答案** | 字串比對、數字驗證 | 文章品質、設計評分 |
| **無標準答案** | 規則型檢查（如字數限制） | 風格一致性、專業度 |

### 組件級評估
除了評估整個系統，也應**隔離單一組件**獨立評估，加速優化迭代。

## 5️⃣ 規劃與多代理系統 (Planning & Multi-agent)

### 規劃 (Planning)
不再寫死步驟，而是提供工具集，讓 LLM 生成**逐步計畫**。

#### 程式碼規劃 > JSON 規劃
讓 LLM 直接寫程式碼來規劃步驟，通常表現更好且更靈活。

### 多代理系統

#### 角色分工
像人類團隊一樣，指派不同角色協作：
- 🔍 Researcher（研究員）
- 🎨 Designer（設計師）
- ✍️ Writer（撰稿人）
- 👔 Manager（經理）

#### 溝通模式

| 模式 | 描述 | 適用場景 |
|------|------|----------|
| **線性接力** | A → B → C → 結果 | 流程化任務（發票處理） |
| **層級/主管** | Manager 指揮 Workers | 需要協調的複雜任務 |
| **深層層級** | 多層管理者 | 大規模系統 |
| **全對全** | 任意代理互相對話 | 創意協作 |

## 🏢 企業場景拆解範例

### 案例 A：發票處理（線性、低自主）
```
1. [輸入] 接收發票 PDF
2. [工具] API 轉換 PDF → 文字
3. [LLM] 判斷是否為發票？提取欄位
4. [工具] 寫入資料庫
```

### 案例 B：客戶訂單查詢（需判斷）
```
1. [LLM] 讀取 Email → 提取客戶姓名、訂單號
2. [工具] 查詢訂單資料庫
3. [LLM] 撰寫回覆 Email 草稿
4. [工具] 放入佇列等待人工審核 (Human-in-the-loop)
```

## 📋 開發 Checklist

### 設計階段
- [ ] 任務是否可拆成多步驟？（迭代工作流程）
- [ ] 哪些步驟需要外部回饋？（反思模式）
- [ ] 需要哪些工具來擴展能力？（工具使用）
- [ ] 如何量化每個組件的表現？（評估矩陣）
- [ ] 是否需要多角色協作？（多代理系統）

### 開發階段
- [ ] 快速原型 > 完美設計
- [ ] 執行後收集 traces
- [ ] 錯誤分析量化到組件級
- [ ] 迭代優化高頻錯誤組件

### 驗證階段
- [ ] 客觀測試（程式碼驗證）
- [ ] 主觀測試（LLM-as-a-judge）
- [ ] 組件級測試 + 整體測試
- [ ] Human-in-the-loop 審核關鍵輸出
