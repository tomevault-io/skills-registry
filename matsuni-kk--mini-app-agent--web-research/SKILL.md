# Web Research Workflow

要件定義や技術選定において、既存知識で確定できない情報をWeb検索で検証・補完する。

## Instructions

### 1. 検索トリガーの判断

以下の場合にWeb検索を実施する：
- ユーザーが明示的に「調べて」「検索して」と依頼した場合
- 技術情報の最新性が不明な場合（API仕様、ライブラリバージョン等）
- 要件の実現可能性が不確定な場合
- 外部サービスの仕様・制限を確認する必要がある場合
- 「〜できるか？」「〜は可能か？」という質問への回答に自信がない場合

### 2. 検索実行

#### WebSearchツールを使用
```
WebSearch: クエリを指定して検索
- 日本語・英語両方で検索を試みる
- 公式ドキュメントを優先する
- 最新の日付のソースを重視する
```

#### WebFetchツールを使用
```
WebFetch: 特定URLの内容を取得・分析
- 公式ドキュメントのURLが判明している場合
- 検索結果から詳細を確認したい場合
```

### 3. 検索クエリの設計

効果的なクエリ例：
- API仕様: `{サービス名} API documentation 2024`
- 制限確認: `{サービス名} limits pricing free tier`
- 実装方法: `{技術名} {やりたいこと} example`
- 比較: `{A} vs {B} comparison`
- トラブル: `{エラーメッセージ} solution`

### 4. 結果の整理

検索結果は以下の形式で報告：

```markdown
## 検索結果サマリー

### 質問
{調査対象の質問}

### 回答
{検索結果に基づく回答}

### 根拠
- [ソース1のタイトル](URL) - 要点
- [ソース2のタイトル](URL) - 要点

### 確度
- 高: 公式ドキュメントで確認済み
- 中: 複数の信頼できるソースで一致
- 低: 情報が古い/単一ソースのみ

### 注意点
{最新情報の確認が必要な点、矛盾する情報など}
```

### 5. 検索できない場合

Web検索が利用できない環境では：
- 既存知識で回答可能な範囲を明示
- 「未確認」「要検証」と明記
- ユーザーに自身での確認を推奨

## 使用ツール

| ツール | 用途 |
|--------|------|
| WebSearch | キーワード検索、最新情報の取得 |
| WebFetch | 特定URLの内容取得・分析 |

## 検索対象の優先度

1. **公式ドキュメント** - 最も信頼性が高い
2. **公式ブログ・リリースノート** - 最新情報
3. **Stack Overflow** - 実装例・トラブルシューティング
4. **技術ブログ** - 実践的な知見
5. **フォーラム・Issue** - エッジケース情報

## 注意事項

- 検索結果は必ずソースURLを付記する
- 古い情報（2年以上前）は注意を促す
- 公式と非公式の情報を区別する
- 矛盾する情報がある場合は両方を提示する
