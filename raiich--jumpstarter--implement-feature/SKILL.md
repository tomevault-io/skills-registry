# Implement Feature

Design Doc とテストケース設計を入力として、TDD 実装を行うスキルです。

## 前提

- 入力は以下のいずれか:
  - `design-feature` スキルの成果物: `design.md`
  - `design-feature` + `design-tests` スキルの成果物: `design.md` + `tests.md`
  - ユーザーからの直接入力
- 小さい単位で TDD サイクルを回す

## タスクの粒度

- 1タスク = 1つの関数・クラス・メソッド程度
- 1タスク = テストケース1〜5個程度

## フロー

### [入力の確認]

#### 1. 入力の確認

`design.md` があれば読み込む。`tests.md` もあれば読み込む。
いずれもない場合はユーザーの入力から要件を把握する。
不明点があればヒアリングする。

**ツール**: Read, AskUserQuestion

### [実装フェーズ] — タスクごとに繰り返し

#### 2. 実装タスクリスト作成

Design Doc とテストケースに基づいてタスクを作成。
Design Doc に ❓（仮定）マークの項目がある場合、該当タスクの実装前にユーザーへ確認する。

**ツール**: TaskCreate

#### 3. TDDサイクル実施

**ツール**: TaskUpdate でタスクを in_progress に設定

各タスクについて以下を実施：

**3.1. テストコード作成（基本パターン）**
- `tests.md` がある場合: テストケース設計に基づいてテストコードを実装
- ない場合: Design Doc またはユーザー入力に基づいてテストケースを設計・実装

**3.2. 機能実装（基本パターン）**
- テストが通るように機能を実装

**3.3. テスト実行と失敗時の対応**

**ツール**: Bash（テスト実行コマンドはプロジェクトの settings に従う）

失敗時は**修正に入る前に原因分析**を行う:

**原因分析（必ず修正前に実施）:**
1. エラーの直接原因を特定する
2. 直接原因のさらに上流を探る — なぜそうなったか？症状の出た場所が本当の原因か？
3. 修正候補を複数検討し、最適なアプローチを選ぶ:
   - この修正は場当たり的でないか？根本原因に対処しているか？
   - 他のテストや機能に影響しないか？
   - 設計レベルの見直しが必要ではないか？

**原因分析の結果に基づく対応:**
- **実装のバグ** → 原因箇所（症状の箇所とは限らない）を修正し再テスト
- **テスト設計のミス** → テストケース設計を見直し、tests.md を更新
- **設計の問題** → design.md に立ち戻り、設計を見直してからやり直す
- **判断不可** → ユーザーにヒアリング

**修正が2回失敗したら**: 同じアプローチを繰り返さず、別の原因仮説やアプローチを検討する。3回失敗したらユーザーに相談する

**3.4. リファクタリング**
- テストが通った後、実装コードを整理（重複排除、命名改善、構造の簡素化など）
- リファクタリング後にテストを再実行し、既存テストが通ることを確認

**3.5. タスク完了**

**ツール**: TaskUpdate で completed に設定、TaskList で次タスク確認

**レビュー粒度**:
- 軽微なタスク: まとめてレビュー可
- 重要なタスク: 個別レビュー

## 自己レビュー観点

### テストコード
- テストケース設計との整合性
- 正常系・異常系・エッジケースの網羅性
- テストの粒度は適切か
- Arrange-Act-Assert が明確に分離しているか
- DAMP > DRY: ヘルパーで隠しすぎず、各テスト内で何をテストしているかが読み取れることを優先
- テスト名だけでは内容が不十分な場合はコメントで補足する

### 実装コード
- 過剰な実装をしていないか
- セキュリティ上の懸念はないか

観点は内容に応じて調整。不明確な場合はユーザーにヒアリング。
