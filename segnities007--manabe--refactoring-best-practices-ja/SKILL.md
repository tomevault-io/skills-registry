# リファクタリング・ベストプラクティス統合スキル（日本語）

## 使うタイミング

- 「ベストプラクティスを外部調査しつつリファクタリングしてほしい」と依頼されたとき
- 設計負債はあるが、仕様逸脱や回帰を避けて段階的に改善したいとき
- 内部設計書（要件/設計/ADR）を更新しながら実装を改善したいとき

## ゴール

- 外部一次情報に根拠づけられた改善方針を作る
- 実装差分を最小化しつつ、可読性・保守性・検証容易性を上げる
- 内部ドキュメントと実装を同期し、再レビュー可能な状態にする

## 入力として必ず確認するもの

1. 対象コード（変更候補ファイル）
2. SoT文書（`doc/decisions/` と関連 `doc/notes/`）
3. 既存テストと実行コマンド（最低1つ）
4. 外部根拠（公式ドキュメント/言語公式/フレームワーク公式）

## 実行ワークフロー

1. スコープ固定
- 対象機能、対象外、非機能制約（性能/運用/納期）を明記する。
- 互換性条件（UI挙動、公開API、データ形式）を固定する。

2. 現状測定（内部）
- `rg` で重複・巨大関数・命名不整合・TODO/TBD を抽出する。
- 変更候補を「安全に分割できる単位」で列挙する。

3. 外部ベストプラクティス調査（一次情報優先）
- 公式一次情報を優先し、主要判断ごとに最低1根拠を持つ。
- 複数候補がある場合はトレードオフ（採用/不採用/理由）を残す。
- URL と確認日（絶対日付）を記録する。

4. 方針設計（内部×外部統合）
- 各リファクタリング項目に「目的/影響/リスク/ロールバック」を付ける。
- 先に SoT を必要最小限更新し、下流文書へ同期する。

5. 実装（小さく刻む）
- 1論点=1差分で進める。
- 振る舞い変更を伴う場合は明示し、テストで保護する。
- 仕様変更を伴わないリファクタリングでは挙動等価を維持する。

6. 検証
- 変更対象モジュールのテスト/ビルドを実行する。
- 失敗時は差分を戻すのではなく原因を切り分け、最小修正で再実行する。

7. 記録
- 外部根拠リンク、内部参照ファイル、実施コマンド、残課題を報告する。
- 次反復での改善候補（優先度付き）を残す。

## ベストプラクティス根拠の優先順位

1. 公式ドキュメント（言語・フレームワーク・標準ライブラリ）
2. 公式スタイルガイド/推奨アーキテクチャ
3. 標準化団体・一次仕様
4. 上記が不足する場合のみ、実績ある技術記事（補助根拠として扱う）

参照先候補は `references/source-priority.md` を使う。
実務で確定した実装規約は `references/best-practices-canon.md` を一次参照とする。

## Canon 採用審査（必須）

`best-practices-canon.md` に追加する規約は、以下をすべて満たす場合のみ採用する。

1. Evidence A/B 判定
- `A`: 公式一次情報 + リポジトリ内で再現済み（テスト/ビルド成功）
- `B`: 公式一次情報のみ（再現未実施）。Canon へは原則追加せず、`playbook` に一時記録する。

2. 反証可能性
- 規約に最小検証コマンドと失敗条件（何が起きたら無効か）を明記する。

3. 差分実証
- 当該規約が、今回の実装差分またはテスト差分に直接効いていることを示す。

4. ドリフト管理
- 既存規約と重複/矛盾を判定し、`採用/不採用/保留` 理由を残す。

## 出力フォーマット

- 対象レベル（Correctness / Testability / Operability / Stress / Decision Pressure）
- 変更方針（3行以内）
- 外部根拠（URL + 確認日 + 採用判断）
- 更新ファイルと更新理由
- 検証結果（実行コマンドと結果）
- 残課題（ブロッカー付き）
- 次反復の入口条件

## ガードレール

- 大規模一括リライトは禁止。追跡可能な最小差分を優先する。
- 公式根拠が不十分な提案は「仮説」と明示し、断定しない。
- テスト未実行のまま完了扱いにしない（実行不可なら理由を明記）。
- 既存の内部 SoT と矛盾する場合は、先に文書側の整合方針を決める。

## 自己改善ループ（使うたびにSkillを強化）

各実行の最後に、以下を必ず更新する。

1. ベストプラクティス本体更新（最優先）
- 再利用可能な知見を `references/best-practices-canon.md` に規約として追加/更新する。
- 記法は「Rule / Why / Apply when / Do / Do not / Verify / Sources」で統一する。
- 各実行で Canon の差分を必ず作る。差分が不要と判断した場合は「不要理由（重複/不十分/保留）」を結果に明記する。
- Canon へ追加する規約は `Evidence A` を満たすもののみとする（`B` は `playbook` 扱い）。

2. パターン抽出
- 一時的または文脈依存の知見のみ `references/refactoring-playbook.md` に追記する。
- 同種パターンが2回以上成立したら Canon へ昇格する。

3. Skill本体反映
- Canon に追加した内容のうち、常に有効なものだけを `SKILL.md` のワークフローへ反映する。
- 一時的・環境依存の知見は `SKILL.md` に直接入れず `references/` に留める。

4. ドリフト防止
- 新規追記時に、既存ルールとの矛盾を確認する。
- 矛盾がある場合は「採用/不採用/保留」を明記してから更新する。

5. 学習ログ（補助）
- 実行履歴が必要な場合のみ `references/learning-log.md` に追記する。
- ログは判断経緯の補助であり、ベストプラクティス本体ではない。

## 実行後チェックリスト（必須）

- [ ] 外部根拠は一次情報を優先し、URLと確認日を記録した
- [ ] 実装差分に対してテスト/ビルドを実行した
- [ ] `references/best-practices-canon.md` に規約を追加/更新した
- [ ] Canon を更新しない場合は、その理由（重複/不十分/保留）を明記した
- [ ] Canon 追加規約が `Evidence A`（公式一次情報 + リポジトリ再現）を満たすことを確認した
- [ ] 文脈依存知見のみ `references/refactoring-playbook.md` に反映した（該当時）
