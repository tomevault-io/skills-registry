# TDDテスト生成

間違った理由（構文エラー、間違ったインポート）で失敗するテストは有効なRedではない。まずテストを修正し、失敗が意図した振る舞いのギャップと一致することを確認する。

## RGRCサイクル

| フェーズ | 目標           | ルール                      |
| -------- | -------------- | --------------------------- |
| Red      | 失敗テスト     | 失敗理由を確認              |
| Green    | テストをパス   | "罪を犯してよい" - dirty OK |
| Refactor | クリーンコード | テストをグリーンに保つ      |
| Commit   | 状態を保存     | 全チェックをパス            |

## Baby Steps (2分サイクル)

30秒: 失敗テスト作成 → 1分: パス → 10秒: テスト実行 → 30秒: 小さなリファクタ →
20秒: グリーンならコミット

## テスト設計

| 技法     | 用途                     | 例                     |
| -------- | ------------------------ | ---------------------- |
| 同値分割 | 同じ振る舞いをグループ化 | 年齢: <18, 18-120      |
| 境界値   | 境界をテスト             | 17, 18, 120, 121       |
| 決定表   | 複数条件ロジック         | isLoggedIn × isPremium |

## モックの落とし穴

| アンチパターン                     | 問題                                          | 代わりに                                         |
| ---------------------------------- | --------------------------------------------- | ------------------------------------------------ |
| モックの呼び出しをアサート         | モックの動作をテスト、コンポーネントではない  | 観察可能な出力や副作用でアサート                 |
| テスト専用のプロダクションメソッド | テストアクセスのためにプロダクションAPIを汚染 | テストユーティリティに抽出またはpublic APIを使用 |
| 理解前にモック                     | 実際の依存関係の動作を隠す                    | まず依存関係を理解してからモック                 |
| 部分的なモック構造                 | 欠落フィールドが偽のパスを生む                | 完全な実際のAPI構造をミラー                      |
